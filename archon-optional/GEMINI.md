# Gemini 适配层 - Archon 非正式智能体

## 角色定位

Gemini 在 Archon 系统中担任**非正式智能体 (Informal Agent)**，负责：
- 生成详细的自然语言数学证明
- 填补非正式证明中的 gap
- 提出替代证明策略
- 数学推理和验证

## 为什么选择 Gemini

根据 Archon 论文："根据经验，Gemini 在这三个模型中往往能提供最强的数学性能。"

## 工作模式

### 输入
- 定理陈述（自然语言或形式化）
- 已有的非正式证明片段
- 形式化障碍描述（如"Mathlib 缺少路径积分"）

### 输出
- 逐步的无 gap 自然语言证明
- 替代证明路径建议
- 关键引理识别

### 调用时机
1. Lean 智能体遇到数学 gap 无法独立填补
2. 规划智能体判断需要"详细非正式支持"
3. 规划智能体判断需要"非正式路线更改"

## 技术实现

### API 调用方式

使用现有的 `mcp__multi-ai__ask` 工具：

```bash
# 通过 MCP 调用 Gemini
model: gemini-3.1-pro-preview
prompt: <数学问题>
```

### 增强能力

**检索能力：**
- 通过 WebSearch 查找标准定理
- 访问 arXiv/MathSciNet（通过 `~/ai/data/keys/` 中的 API）

**自我反思：**
- 生成证明后自我检查逻辑 gap
- 识别可能的形式化障碍

## 与 Claude 的协同

| 任务 | Claude (Lean 智能体) | Gemini (非正式智能体) |
|------|---------------------|---------------------|
| 形式化代码生成 | ✓ 主责 | ✗ |
| 数学推理 | ○ 辅助 | ✓ 主责 |
| 策略规划 | ✓ (规划智能体) | ○ 提供数学视角 |
| 库 API 使用 | ✓ | ✗ |
| 证明 gap 填补 | ○ 尝试 | ✓ 主责 |

## 示例工作流

### 场景 1: 填补证明 gap

```
[Lean 智能体] → 卡在引理 X，无法证明
[规划智能体] → 判断需要详细非正式支持
[规划智能体] → 调用 Gemini: "为引理 X 生成逐步证明"
[Gemini] → 返回 5 步详细证明
[规划智能体] → 将证明传递给 Lean 智能体
[Lean 智能体] → 按步骤形式化
```

### 场景 2: 路线更改（FirstProof 问题 4 实例）

```
[Lean 智能体] → 儒歇定理需要路径积分，Mathlib 中不存在
[规划智能体] → 判断需要非正式路线更改
[规划智能体] → 调用 Gemini: "不使用儒歇定理，证明多项式根连续性"
[Gemini] → 提出 Vieta 公式方法
[规划智能体] → 指示 Lean 智能体: "使用 Vieta 公式"
[Lean 智能体] → 按新路线形式化
```

## 质量控制

### 输出验证
- 证明步骤是否逻辑连贯
- 是否存在隐含假设
- 是否适合形式化（避免过于依赖直觉）

### 失败处理
- 如果 Gemini 输出含糊，要求重新生成并明确指出 gap
- 如果多次失败，考虑切换到 GPT-5.4 作为备用

## 成本估算

Gemini 3.1 Pro Preview 调用成本远低于 Claude Opus，适合作为高频数学推理后端。

## 注意事项

1. **上下文隔离**: Gemini 不应看到 Lean 代码细节，只处理数学层面
2. **语言一致性**: 输入输出均使用中文（除非数学符号）
3. **避免过度依赖**: 简单的数学推理应由 Claude 直接处理
