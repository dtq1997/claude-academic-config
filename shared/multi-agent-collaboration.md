# 多 AI 共管协议

更新时间：2026-03-06
适用范围：`/Users/dtq1997/ai/**`
参与者：Codex、Claude、Gemini

## 默认前提

- 所有任务默认属于三家 AI 共管系统。
- 任一时刻允许单家主执行，不要求三家同时在线。
- 任何关键进展、阻塞、决策、工具入口都必须保持 handoff-safe。

## 强制原则

- 私有对话上下文不能作为唯一事实来源。
- 长期有效的信息必须落盘到共享文件。
- 新增工具或自动化时，优先采用 vendor-neutral 的脚本、CLI、JSON、Markdown。
- 若某项改动会降低另外两家接管能力，先提示风险，再走最小可逆路径。

## 推荐分工

- Planner：定义目标、拆任务、维护决策依据。
- Executor：改文件、跑脚本、推进主实现。
- Reviewer：复核结果、发现遗漏、提出回归检查。
- Fallback：某家不可用时，另一家按共享状态直接接管。

同一任务可由一家的多个角色合并承担；不强制三方同时参与。

## 共享状态最小要求

以下信息应优先写入 `~/ai/**`：

- 项目目标与当前状态
- 下一步动作
- 阻塞与风险
- 关键决策及理由
- 可复用的工具入口

## 工具接入原则

- 优先级：本地脚本/CLI > 统一 wrapper > MCP > 单 agent 私有集成。
- API key 统一保存在 `~/ai/data/keys/` 或 `~/ai/config/secrets.json`。
- 调用方应尽量通过统一入口查询，而不是把 endpoint、header、密钥规则硬编码到 prompt。

## 执行定义

“考虑三家共管”至少意味着：

- 不把信息只留在单次聊天里
- 不写死只适用于某一家 AI 的操作路径
- 修改状态文件时，保证命名和结构便于其他 AI 读取
- 能复用现有 wrapper 时，不重复造私有调用链

## 冲突处理

- 优先以已落盘规则、项目文档、用户最新明确指令为准。
- 若三家建议不一致，先记录分歧，再按用户目标选择最小风险路径。
- 未验证事实要标记为待验证，不把猜测写成结论。

## 文档审校工作流（大型 LaTeX / 学术文档适用）

> 来源：用户朋友的"三家意见整合指令"，2026-03 适配 [Claude]

### 流程

1. **分头审阅**：用户在各家 CLI 中分别提交同一文档，收集各自审阅意见
2. **整合仲裁**：由任一家（或用户手动）汇总三家意见，按以下结构分类
3. **结果落盘**：整合报告存入项目目录下 `COMPARE/` 文件夹

### 三段式分类

| 类型 | 标签 | 含义 |
|------|------|------|
| 共识 | `[三家共识]` / `[Claude+Gemini]` | 两家以上指出同一问题，整合为一条 |
| 独家 | `[Gemini 独家]` 等 | 仅一家提出，核实后保留 |
| 分歧 | `[意见分歧：Claude 胜出]` 等 | 多家意见矛盾，仲裁方给出裁决和依据 |

### 仲裁原则

- **事实类**（公式、数值、年份、引用）：查原始文献或权威来源，谁对谁赢
- **风格/结构类**：记录分歧，标注各方理由，由用户最终决定
- **某家意见本身有错**：直接驳回并说明，标签 `[驳回：GPT 建议有误]`

### 报告模板（存为 `COMPARE/章节名_三家整合.md`）

```
# 三家意见整合：[章节名]

## 一、共识意见
- **[问题标题]** 位置：L42 | 综合意见：… | 提出者：[三家共识]

## 二、独家意见
- **[问题标题]** 位置：L87 | 意见：… | 提出者：[Gemini 独家]

## 三、分歧裁决
- **[争议点]** Claude：… / GPT：… | 裁决：[依据 + 结论]

## 四、总结
本章修改重心：…
```

### 注意事项

- 不要求每次审校都走此流程；简单修改直接做
- 适用场景：学术综述、论文投稿稿、长文档（>10 页）的系统审校
- 可逐章执行，也可全文一次性整合
