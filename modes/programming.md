# 编程模式

## 自迭代原则

**做完任何修改后，必须自主验证结果，不依赖用户手动测试。**
- 能用工具验证的，立即跑（静态分析、语法检查、grep 确认引用等）
- 验证发现问题 → 当场修复 → 再次验证 → 循环直到通过
- 用户看到的应该是已验证的最终结果，不是需要人工复核的中间产物

**写完代码后立即执行验证，不等用户：**
1. 对修改文件运行静态分析（Python: `radon cc -s -n C`；JS/TS: `npx eslint`；无工具的语言人工审查复杂度）
2. 对新增文件确认行数
3. 不达标 → 当场修复 → 重新检查 → 循环直到通过
4. 全部通过后输出质量报告：

```
📊 质量报告
├─ 修改文件: xxx.py (行数), yyy.py (行数)
├─ 静态分析: 全部通过 ✅ / ❌ 列出问题
├─ 最长新增方法: _xxx() N行 ✅/❌
└─ 状态: ✅ 全部达标 / ❌ 需修复
```

**没有质量报告 = 违规。** 豁免：单文件 ≤5 行改动（typo、配置微调等）无需出报告。

## Bug 修复：同类全扫（不可只修一处）

修一个 bug 时，必须假设同类 bug 存在于其他代码路径。

**强制工作流：**
1. 修当前 bug，确认修复有效
2. 提炼 bug 的模式特征（如：`.after(0,` 在非 root Toplevel 上、`_lib_populate_tree()` 绕过筛选状态）
3. 全盘 grep 该模式，逐个分析每个调用点
4. 一次性修完所有同类 bug，不留漏网
5. 质量报告中列出所有修复点 + 确认安全的调用点

**只修一处 = 未完成。** 用户换个操作触发另一条路径就会复现。

## 日志与调试模式（开发基础设施）

**日志是 AI 自迭代的眼睛——没有日志就是盲调。**

任何非 trivial 的项目都应尽早建立日志基础设施：
- 关键代码路径（数据加载、状态切换、异步回调）必须有日志输出
- 日志应包含：时间戳、函数名、关键参数值、执行结果
- 提供 debug 模式开关（环境变量或配置项），开启后输出详细日志，关闭后静默
- 错误路径必须记录完整 traceback，不能静默吞掉异常
- 异步/多线程场景额外记录线程名和回调链路，方便追踪时序问题

**调试时优先看日志，不要靠猜。** 如果日志不够用，先补日志再调 bug。

## 代码质量红线

- 函数圈复杂度 CC ≤ 25，超过必须拆分
- 新增方法 ≤ 80 行（UI 构建 ≤ 120 行）
- 新增文件 ≤ 800 行
- 循环内逻辑 ≤ 30 行，嵌套 ≤ 3 层
- 筛选链 > 5 条件 → 提取为独立函数
- 禁止复制粘贴 > 10 行
- 先读代码再改代码，不提议未读过的修改
- 最小改动原则，不过度工程
- 不主动加测试、注释、类型标注，除非被要求

## 任务过载保护

单轮不超过 2 个独立功能需求。超过时主动拆分并告知用户。

## 中断保护

多文件联动修改被打断时，必须警告风险并建议完成或回滚。

## 节省 Token

- 渐进式阅读（强制）：禁止一次性通读大文件，必须先 Grep/Glob 定位再用 offset+limit 精确读取，单次不超过 500 行
- 复杂重构必须先进 plan mode
- 增量编辑，用 Edit 工具逐处修改，禁止整体重写文件
- AI 有权提出独立判断和建议，用户期望主动性而非被动执行

## 屏幕控制能力（GUI 验证用）

- MCP 工具：`mcp-pyautogui-server`（pip install mcp-pyautogui-server）
- 能力：截图、鼠标点击、键盘输入、滚动
- 用途：修改 GUI 代码后，可自主截图验证界面是否正确，无需用户手动反馈
- macOS 需授予"辅助功能"权限（系统设置 → 隐私与安全 → 辅助功能）
- 工作流：改代码 → 启动程序 → 截图 → 看到问题 → 自行修复 → 再截图确认

## 编程经验教训

- **脚本必须保留错误可见性**：可能闪退的脚本必须在关键调用后加阻塞（`pause`/`read`），确保报错可见

## 网络/异步代码检查清单（mobile-bridge 48 bug 教训）

写涉及网络请求、异步任务、多线程的代码时，逐条检查：

**每个 fetch/HTTP 请求必须有：**
- AbortController + setTimeout 超时（前端 8-15s，后端按业务定）
- catch 里清除 timeout timer
- 区分超时 vs 网络错误 vs 业务错误的提示文案
- 非 JSON 响应的防御（检查 content-type 再 `.json()`）

**每个错误路径必须重置 UI 状态：**
- 用状态机或统一的 `resetToIdle()` 收敛散落的状态变量（按钮 disabled、状态文字、loading 标志）
- 禁止在 N 个 catch/if 分支里手动重复 `btn.disabled=false; status='就绪'; sending=false`
- 新增错误路径时，先确认状态重置是否完整

**异步任务必须有兜底完成机制：**
- 后台线程/Promise 的任务必须用 `try/finally` 保证标记为 done，不能只在 happy path 标记
- 子进程 stdin 写入要 catch BrokenPipeError
- 轮询要有最大次数/超时上限，不能无限等

**资源生命周期配对检查：**
- blob URL：createObjectURL ↔ revokeObjectURL，注意 revoke 时机不能早于 DOM 使用
- 临时文件：创建 ↔ 删除，用 finally 保证清理
- subprocess：stdin.close() 必须在 finally 里

**同类全扫（与上方规则联动）：**
- 修一个 fetch 缺 timeout → grep 所有 fetch 调用，一次性全补
- 修一个错误路径缺状态重置 → 检查所有错误路径
