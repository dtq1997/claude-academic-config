# 写作方法论

## 〇、LLM 写数学的已知缺陷与防护（读本文件前先读这一节）

LLM 写数学综述有**系统性的失败模式**，不是偶发 bug，是结构性风险。以下每一条都必须有对应的防护措施。

### 已知失败模式

| 编号 | 失败模式 | 说明 | 防护 |
|------|---------|------|------|
| F1 | **假设条件漂移** | 悄悄丢条件："对所有 $s$" vs "Re($s$)>1"；"$\|q\|<1$" 消失；"一致" vs "逐点" 混用 | 每个定理写完后逐条检查假设 |
| F2 | **幽灵引用** | 论文真实存在，但不包含我们声称的结论 | 引用内容验证（见下） |
| F3 | **证明草稿陷阱** | 看起来完整的证明，跳过了唯一困难的一步（解析延拓的合法性、围道积分的增长估计、误差项控制） | 证明中每一步标注依据；跳步必须显式声明 |
| F4 | **近似对象混淆** | Ramanujan 求和 ≠ zeta 正则化 ≠ Abel 求和（虽然某些情况给出相同数值）；Stirling 第一类 ≠ 第二类；$B_1 = \pm 1/2$ | 全局术语表 + 首次出现给精确定义 |
| F5 | **符号碰撞** | $E_n$ 在第 9 章是 Euler 数，在第 4 章是 Euler 求和参数，在第 14 章是 Eisenstein 级数 | 全局符号注册表（见下） |
| F6 | **编造引用** | 虚构论文、虚构页码、把真论文的结论张冠李戴 | 每条引用必须 API 验证存在性 |
| F7 | **过度泛化** | 从一个例子跳到 "对所有 $n$ 成立"，不检查边界情况 | 计算验证 + 反例搜索 |
| F8 | **认识论扁平化** | 把猜想写成定理，把 folklore 写成已证结果，把物理论证写成数学证明 | 认识论分级标记（见下） |
| F9 | **历史归属错误** | 误归 Ramanujan（实际是 Hardy 或更早）、压缩时间线、把猜想当已证 | 历史事实双源验证 |
| F10 | **采样偏差** | 偏向 arXiv 上可见的现代文献，低估非数字化的经典著作 | 必引基线书目（见下） |

### 零信任原则

> **模型记忆中的一切数学"事实"均视为未验证，直到有外部来源确认。**

这不是建议，是强制规则。即使是 "$\zeta(-1) = -1/12$" 这种看起来毋庸置疑的事实，写入正式文件时也必须注明来源。

## 一、认识论分级标记

本项目涉及大量处于不同严谨性层次的数学内容。**每个非平凡声明必须标注认识论级别**。

| 标记 | 含义 | 使用场景 |
|------|------|---------|
| `[定理]` | 有完整严格证明的结果 | 经典定理、已证猜想 |
| `[命题/引理]` | 有证明但重要性较低的辅助结果 | 技术性结论 |
| `[猜想]` | 被广泛相信但未证明 | Ramanujan 猜想（Deligne 之前） |
| `[形式论证]` | 形式操作给出的结论，严格性待确立 | Ramanujan 风格推导、umbral 操作 |
| `[启发式]` | 直觉性论证，不声称严格性 | 物理论证、概率启发 |
| `[数值证据]` | 仅有数值支持 | 未证明的恒等式 |
| `[folklore]` | 专家圈内"众所周知"但找不到明确出处 | 必须标注"据 [X] 称…" |
| `[物理论证]` | 物理学中使用的推导，数学严格性未建立 | 重整化、路径积分论证 |

**规则：**
- 正文中每个定理/命题首次出现时必须带标记
- `[形式论证]` 必须同时指出严格化的路径或障碍
- 禁止将 `[猜想]` 或 `[folklore]` 不加标记地当作已证结果使用
- 后续章节引用时如果认识论级别发生变化（如猜想被证明），必须说明

## 二、调研数据源

每一章写作前**必须主动接入数据库**做系统性调研和核实，不能只靠模型内部知识。

### 一级数据源（每章必查）

| 数据库 | 用途 | 调用方式 |
|--------|------|---------|
| **zbMATH Open** | 数学专属评审库，MSC 分类检索，获取专家 review | `curl "https://api.zbmath.org/v1/document/_search?search_string=ti%3A%22...%22"` 注：参数需 URL 编码 |
| **Semantic Scholar** | 论文摘要、引用网络、相关推荐，追踪引用链 | `curl -H "x-api-key: $S2_API_KEY" "https://api.semanticscholar.org/graph/v1/paper/search?query=..."` 注：无 key 时严重限速，key 见 `~/ai/data/keys/` |
| **arXiv** | 预印本搜索，获取最新进展的 PDF | `curl "https://export.arxiv.org/api/query?search_query=..."` |
| **OEIS** | 整数序列核实（Bernoulli 数、分拆数等） | `curl "https://oeis.org/search?fmt=json&q=..."` |
| **LMFDB** | L-函数、模形式、椭圆曲线的具体数据核实 | `curl "https://www.lmfdb.org/api/..."` |

### 二级数据源（按章节需要）

| 数据库 | 适用章节 | 调用方式 |
|--------|---------|---------|
| **DLMF (NIST)** | 特殊函数、渐近展开（第5,7,11,22章） | 网页抓取（各家用自己的工具） |
| **nLab** | 范畴论视角、Hopf 代数（第10,19章） | 网页抓取 |
| **INSPIRE-HEP** | 物理应用文献（第20章） | `curl "https://inspirehep.net/api/literature?q=..."` |
| **CrossRef** | DOI 元数据、引用格式生成 | `curl "https://api.crossref.org/works/DOI"` |
| **OpenAlex** | 大规模文献趋势、作者网络 | `curl "https://api.openalex.org/works?search=..."` |
| **Unpaywall** | 查 OA 全文 PDF | `curl "https://api.unpaywall.org/v2/DOI?email=..."` |

### 三级数据源（补充）

| 数据库 | 用途 |
|--------|------|
| **Numdam** | 法国数学期刊历史文献（Borel, Cesàro 原始论文） |
| **Project Euclid** | 数学/统计期刊 |
| **Wikipedia + MacTutor** | 历史人物生平、年份核实（网页抓取，需交叉验证） |
| **Google Scholar** | 补充搜索（网页搜索） |

### 调研规则

1. **定理核实**：任何定理/命题的精确陈述必须追溯到原始论文或权威教材（Hardy, Rota, Berndt 等），不依赖模型记忆
2. **数值核实**：涉及具体数值（Bernoulli 数、分拆数、zeta 函数特殊值等）必须用 OEIS 或 LMFDB 验证
3. **历史事实**：年份、人物、优先权争议必须至少两个独立来源交叉验证
4. **引用完整性**：每个 `[Author, Year]` 引用必须有对应的 DOI 或可追溯的出处
5. **引用内容验证**：高影响引用（被 ≥3 处使用的文献）不仅要确认论文存在，还要确认它确实包含我们声称的结论（防 F2）
6. **多模型验证**：关键数学论断由至少两家 AI 交叉验证，标注验证方（如 `[Claude + GPT]`、`[Gemini + Codex]`）
7. **arXiv 版本锁定**：引用 arXiv 预印本时记录版本号（v1/v2/...），优先引用正式发表版

## 三、公式与证明的处理策略

### LLM 处理数学公式的能力分层

| 任务类型 | 可靠度 | 处理方式 |
|---------|--------|---------|
| 概念叙述、历史、动机 | 高 | 直接写，正常核实 |
| 陈述标准定理、写 LaTeX 公式 | 中高 | 从权威来源（Hardy/Berndt/DLMF）抄写精确陈述，SymPy 验证数值一致性 |
| 简单推导（≤5 步） | 中 | 写出后逐步验证，注意正负号和系数 |
| 复杂多步证明 | 低 | **不要让模型从头推导**，改用思路级或引用级处理 |
| 符号计算（展开、化简） | 低 | 交给 SymPy/mpmath，模型只负责解释结果 |

核心原则：**本项目是综述，模型的角色是组织者和解释者，不是证明机器。** 定理从权威来源获取精确陈述，证明从文献获取，计算交给 CAS。模型负责将这些素材编织成连贯的叙事。

### 证明的三级取舍

| 级别 | 标准 | 处理方式 |
|------|------|---------|
| **展示级** | 精彩 + 短（正文 ≤1 页）+ 体现核心主线（形式操作的力量） | 正文完整写出。**每一步标注依据**，关键等式 SymPy 验证。推导来源标注原始文献。 |
| **思路级** | 重要但较长，或技术性强 | 给证明思路（关键步骤 + 核心 idea），标注"完整证明见 [X, Ch.Y, Thm Z]" |
| **引用级** | 标准结果 / 太长太技术 / 超出 LLM 可靠处理范围 | 仅陈述定理 + "证明见 [X, Ch.Y]"，不尝试写证明 |

### 判断原则

一个证明是否写入正文，**唯一的标准是它是否展示了本综述关心的核心现象**：
- ✅ 形式操作如何工作、为什么"错误的推理给出正确的结果" → 展示级
- ✅ 不同方法殊途同归的对比（同一结果的多条路径） → 展示级
- ❌ 纯技术性地建立某个结果，不体现方法论 → 思路级或引用级
- ❌ 需要重型分析机器（复杂估计、深层代数几何） → 引用级

### 展示级证明的预选清单

以下推导/证明是正文中展示的候选（写作时可增删，但需说明理由）：

| 推导 | 章节 | 展示理由 |
|------|------|---------|
| Euler 对 $1+2+3+\cdots = -1/12$ 的形式操作 | §6.3 | 核心范例：形式操作的力量和危险 |
| 同一结果的三条路径对比（Euler 形式操作 / Ramanujan 求和 / zeta 正则化） | §6.4 / §17.4 | 核心主线：不同方法为什么给出同一个答案 |
| Umbral 等式 $(B+1)^n = B^n$ 的推导与验证 | §9.2 | Umbral 魔术的原型展示 |
| Abel 求和蕴含 Cesàro 求和 | §4.4 | 简洁优美，说明求和法层级 |
| Ramanujan Master Theorem 的形式推导 | §17.1 | Ramanujan 直觉的巅峰之一 |
| 某个经典恒等式的 umbral 证明 vs 传统证明对比 | §11 / §18 | 展示 umbral 方法的效率 |
| Borel 求和 = Laplace 变换的关系推导 | §5.2 | 揭示 Borel 方法的深层结构 |

### 展示级证明的安全措施

由于 LLM 写证明不可靠（F3），展示级证明必须额外执行：
1. **逐步来源标注**：每一个非平凡等号标注依据（"由 [定理 X.Y]"/"由定义"/"由 SymPy 验证"）
2. **SymPy 逐步验证**：关键中间等式用计算验证，脚本存 `verify/`
3. **原始来源比对**：写完后与权威文献中的推导比对，确保没有跳步或偷换条件
4. **禁止发明证明**：展示级证明必须基于已有文献中的推导，可以重新组织表述，但逻辑链不能自创

## 四、计算验证

关键数学恒等式和数值**必须用 CAS 程序验证**，不能只靠目视检查。符号计算是 CAS 的活，不是 LLM 的活。

### 工具

```bash
# 首选：Python + SymPy（已安装）
python3 -c "from sympy import *; ..."

# 数值验证：mpmath（高精度）
python3 -c "from mpmath import *; mp.dps = 50; ..."
```

**已知环境问题：**
- macOS 上 Python `urllib.request` 对 HTTPS 有 SSL 证书验证问题。`verify/` 脚本中需要调外部 API（如 OEIS）时，用 `subprocess` 调 `curl` 而非 `urllib`
- SymPy 使用 $B_1 = +1/2$ 约定，本项目约定 $B_1 = -1/2$（见 `notation-registry.md`）。偶数下标两种约定一致，奇数下标需手动转换。验证脚本中应注释说明约定差异

**OEIS API 格式说明：**
- `https://oeis.org/search?fmt=json&q=...` 返回 bare JSON array `[{...}]`（不是 `{"results": [...]}`）
- 序列数据在 `data[0]["results"][i]` 中，每条序列的值在 `"data"` 字段（逗号分隔字符串）

### 必须计算验证的场景

| 场景 | 验证方法 |
|------|---------|
| 恒等式（求和、乘积、变换） | 两边分别计算前 10-20 项，高精度比较 |
| 递推关系 | 用递推生成前 N 项，与 OEIS/已知值比较 |
| 渐近展开 | 中等范围（$n = 10, 100, 1000$）数值评估，检查误差项行为 |
| 生成函数 | 展开前 N 项，与显式公式比较 |
| Bernoulli/Euler 数具体值 | 与 OEIS 序列比较 |
| Zeta 函数特殊值 | 用 mpmath 高精度计算 |
| 模形式的 Fourier 展开 | 前若干项与 LMFDB 数据比较 |

### 验证代码的管理

- 验证脚本存入 `verify/` 目录，按章节命名：`verify/ch04_cesaro.py`
- 每个脚本自包含，可独立运行
- 输出 PASS/FAIL + 具体数值
- 正文中引用验证结果时标注 `[已计算验证]`

## 五、增量持久化（最高优先级工程规则）

长任务中 agent 意外中断（网络断、token 耗尽、进程被杀）会导致内存中的一切丢失。**做一点、写一点、存一点**。

### 核心原则

> 任何花费超过 1 分钟获取的信息，必须立刻写入文件。内存中的知识不存在。

### 具体规则

1. **查到一篇文献 → 立刻追加到 `references.md`**，不要攒着
2. **查到一个事实/定理 → 立刻写入 `.research.md`**，哪怕只是一行笔记
3. **写完一个小节 → 立刻保存**，不要等整章写完
4. **每完成一个调研子任务 → 更新 `status.md`** 中的状态描述（可加备注）
5. **发现跨章联系 → 立刻追加到 `cross-refs.md`**

### 文件写入粒度

| 操作 | 写入时机 | 目标文件 |
|------|---------|---------|
| API 返回一批文献 | 立刻 | `references.md` |
| 读完一篇论文/页面提取要点 | 立刻 | `chapters/XX-name.research.md` |
| 核实了一个数值/定理 | 立刻 | `.research.md`（标 `[已核实]`） |
| 计算验证通过 | 立刻 | `verify/chXX_*.py` + `.research.md`（标 `[已计算验证]`） |
| 写完一个小节草稿 | 立刻 | `chapters/XX-name.md`（小节级保存） |
| 发现与其他章节的联系 | 立刻 | `cross-refs.md` |
| 阶段性完成（调研→骨架→初稿） | 立刻 | `status.md` |

### 防丢失检查点

- **调研阶段**：每查完一个数据源就写一次 `.research.md`，不要查完所有再写
- **写作阶段**：每写完一个 `§X.Y` 小节就保存一次正式 `.md`，不要等整章
- **如果即将进行大量搜索**：先把已有笔记写入文件，再开始新搜索
- **上下文变长时**：主动建议 `/compact`，但 compact 前确保所有成果已落盘

### .research.md 格式

增量追加友好的格式：

```markdown
# 第 X 章 调研笔记

## [时间戳] 数据源：zbMATH
- 搜索词：...
- 找到 N 篇相关文献
- 关键发现：...
- 待深入：...

## [时间戳] 数据源：Semantic Scholar
- ...

## [时间戳] 事实核实
- ✅ Cesàro 1890 年提出 (C,1) 求和 —— 来源：zbMATH review de:0022.30403
- ✅ $B_1 = -1/2$ vs $B_1 = +1/2$ 两种约定 —— 来源：DLMF §24.2
- ❌ [原以为 X，实际是 Y] —— 来源：...
- 🔢 [已计算验证] $\sum_{k=0}^{n} B_k = B_n$：SymPy 验证前 20 项通过
```

## 六、全局符号注册表与术语管控

### 符号注册表：`notation-registry.md`

所有在正文中使用的数学符号必须在此注册。**一个符号在全项目中只能有一个含义。**

格式：
```
| 符号 | 含义 | 适用章节 | 首次定义 |
|------|------|---------|---------|
| $B_n$ | Bernoulli 数（$B_1 = -1/2$ 约定） | 全局 | §1.1 |
| $(\text{C}, k)$ | Cesàro 求和法 | 全局 | §4.1 |
| $\Delta$ | 差分算子 $\Delta f(x) = f(x+1) - f(x)$ | 第9-11章 | §9.3 |
| $E_n$ | Euler 数 | 第9,11章 | §9.2 |
| $E_{2k}$ | Eisenstein 级数 | 第14章 | §14.1 |
```

**规则：**
- 同一符号如果在不同上下文有不同含义（如 $E_n$），必须在每次使用时消歧
- 新增符号时先查注册表有无碰撞
- `notation-registry.md` 是只追加文件（多终端安全）

### 已固定的全局约定

| 约定 | 选择 | 备注 |
|------|------|------|
| Bernoulli 数符号 | $B_1 = -1/2$ | DLMF 约定 |
| 模参数 | $q = e^{2\pi i \tau}$ | 除非声明使用 nome $q = e^{\pi i \tau}$ |
| Fourier 变换 | $\hat{f}(\xi) = \int f(x) e^{-2\pi i x\xi} dx$ | 如果用到 |
| $\log$ 分支 | 主值，$\arg \in (-\pi, \pi]$ | |
| 求和法符号 | $(\text{C}, k)$, $(\text{A})$, $(\text{B})$, $(\text{E}, q)$, $\mathfrak{R}$ | |
| 形式幂级数环 | $\mathbb{K}[[x]]$ | |
| Zeta 函数 | $\zeta(s)$ = Riemann, $\zeta(s, a)$ = Hurwitz | |

### 受控术语表

以下术语在本项目中有精确含义，禁止混用：

| 术语 | 精确含义 | 常见误用 |
|------|---------|---------|
| Ramanujan 求和 | 通过 Euler-Maclaurin 公式定义的 $\mathfrak{R}$-求和 | ≠ zeta 正则化（虽然数值可能相同） |
| 正则化 (regularization) | 赋予发散表达式有限值的系统方法（需指明具体方法） | 不可无限定词地使用 |
| 求和法 (summation method) | 满足线性和稳定性的将发散级数映射到数值的方法 | ≠ 任意赋值 |
| 形式操作 | 不依赖收敛性的代数操作 | ≠ 严格证明 |
| umbral 等价 | 在特定线性泛函下的等价（需指明泛函） | ≠ 一般等式 |

## 七、结果依赖图与错误传播

### 依赖追踪：`dependency-dag.md`

记录章节/定理之间的逻辑依赖关系。**如果 A 的正确性依赖于 B，那么 B 出错时必须审查 A。**

格式：
```
## 核心依赖

Euler-Maclaurin 公式 (§4.1/附录B)
├── Ramanujan 求和的定义 (§6.1)
│   ├── 1+2+3+... = -1/12 的推导 (§6.3)
```

## 九、单章写作流程

每一章按以下阶段推进，**严格按顺序**。每个子步骤完成后立刻写入对应文件。

### 阶段 1：调研（增量写入 `.research.md` + `references.md`）
- **系统性文献检索**：zbMATH → 写入 → Semantic Scholar → 写入 → arXiv → 写入
- **必引基线检查**：对照第七节书目，确认哪些与本章相关
- **核心文献精读**：每读完一篇，立刻提取要点写入 `.research.md`
- **数据核实**：每核实一条，立刻标记 ✅/❌ 写入 `.research.md`
- **计算验证**：关键恒等式写验证脚本，存入 `verify/`
- **交叉验证**：关键事实由其他 AI 验证（或标注 `[待交叉验证]`），结果立刻写入
- 每个文献条目发现时就追加到 `references.md`
- **每次会话结束前更新 `status.md` 时间戳**（多 AI 协调的超时规则依赖此时间戳，见 `parallel.md`）

### 阶段 2：骨架（写入正式 `.md` 文件头部）
- 基于调研，写出小节结构 + 每小节要点
- 确定关键定理/命题/例子的选取，**每个标注认识论级别（见 §一）和证明级别（展示/思路/引用，见 §三）**
- 检查符号注册表，新符号立刻注册
- 标注交叉引用点 → 写入 `cross-refs.md`
- 更新 `dependency-dag.md`
- 保存，标 `[骨架]`，更新 `status.md`

### 阶段 3：正文（逐小节写入）
- **每写完一个 §X.Y → 立刻保存文件**
- 每个定理/命题带认识论标记
- 每个非平凡公式要么标 `[已计算验证]` 要么标 `[待验证]`
- 所有引用实时追加到 `references.md`
- 全部小节完成后标 `[初稿]`，更新 `status.md`

### 阶段 4：审校
- **F1 检查**：逐个定理检查假设条件是否完整（最常见错误）
- **F2 检查**：高影响引用的内容验证
- **F4/F5 检查**：术语和符号与全局注册表一致
- **依赖检查**：本章是否正确引用了前置结果的条件
- 检查无 `[待验证]` 遗留
- 补充遗漏的参考文献
- 标 `[已审校]`，更新 `status.md`

## 十、写作规范

### 数学严谨性
- 定理/命题必须给出精确条件，不含糊（防 F1）
- 区分认识论级别（防 F8）：`[定理]` vs `[形式论证]` vs `[猜想]` 等
- Ramanujan 风格的形式操作要同时给出：(a) 原始形式推导 (b) 现代严格化路径
- 历史性陈述标注原始出处（防 F9）
- 求和法不同时，必须声明当前使用的方法（防 F4）
- 交换求和/极限/积分顺序时，必须给出合法性的依据或引用

### 叙述风格
- 每章开头：动机段落（为什么关心这个问题）
- 每章结尾：与其他章节的联系 + 进一步阅读指引
- 先具体例子，后一般理论
- 关键转折点（思想跳跃）显式标出
- 每章包含一个"常见陷阱"小节，标出该领域最容易犯的错误

### 交叉引用格式
- 章内：`见 §X.Y`
- 跨章：`见第 N 章 §N.M`
- 前向引用（引用后面的章节）：`将在第 N 章详细讨论`
