# 多 AI 并行协调

## 元要求（最高优先级）

> **本项目由三家 AI（Claude、Gemini、Codex/GPT）共同参与维护。每家 AI 在启动时必须知道这个前提，并以此为基础执行一切操作。**

三家地位对等，不存在"主力"和"辅助"之分。任何一家都可以认领任何章节。

## 核心原则

三家 AI 可以同时工作，但必须：
1. **不写同一个文件**（文件级隔离）
2. **知道彼此在做什么**（通过 `status.md` 协调）
3. **遵循相同的符号和风格约定**（通过 `methodology.md`）
4. **知道其他 AI 可能也在工作**——读共享文件时预期内容可能被其他 AI 更新过

## 各家 AI 的项目说明文件

| AI | 自动读取的文件 | 说明 |
|----|--------------|------|
| Claude | `CLAUDE.md` | Claude Code 启动时自动注入 |
| Gemini | `GEMINI.md` | Gemini CLI 启动时自动注入 |
| Codex/GPT | `AGENTS.md` | Codex CLI 启动时自动注入 |

三个文件各自包含：元要求声明、该 AI 的工具适配指引、指向共享规则的指针。共享规则（`methodology.md`、`outline.md` 等）三家直接读原文件，不重复。

## 协调机制

### 状态文件：`status.md`

所有 AI 共用一个状态文件，作为唯一的协调中心。

格式：
```
| 章节 | 状态 | 执行者 | 最后更新 |
|------|------|--------|---------|
| 01-history | 写作中 | Claude | 2026-03-06 14:00 |
| 09-classical-umbral | 调研中 | Gemini | 2026-03-06 14:05 |
| 06-ramanujan-summation | 待开始 | - | - |
```

- **执行者**列填 `Claude` / `Gemini` / `Codex` 中的一个
- 状态值：`待开始` → `调研中` → `骨架` → `写作中` → `初稿` → `审校中` → `已完成`（审校不通过时回到 `修订中` → `初稿`）

### 工作流

#### 开始一个章节前
1. 读 `status.md`，确认目标章节没有被其他 AI 占用
2. 更新 `status.md`，标记自己的名称和状态
3. 开始工作

#### 章节已被占用时
- 目标章节已被其他 AI 占用且状态活跃（最后更新 < 7 天）：**选择其他章节**
- 最后更新 > 7 天：提醒用户确认是否转交，不得直接覆盖
- 不允许直接覆盖他人的占用状态
- 如需两家协作同一章，用户在主终端拆分为子任务后各自认领

#### 工作过程中
- 只修改自己负责的章节文件
- `references.md` 是共享文件：**追加不删除**，追加时标注章节来源和执行者
- 发现与其他章节相关的内容，记录到 `cross-refs.md`（追加，标注执行者）
- **每次会话工作该章节时更新 `status.md` 的最后更新时间戳**（超时规则依赖此时间戳）

#### 完成时
1. 更新 `status.md` 中的状态
2. 如有交叉引用需要，在 `cross-refs.md` 中记录

### 共享文件的写入规则

| 文件 | 规则 |
|------|------|
| `status.md` | 可改自己认领的行、释放自己占用的行、认领空行。审校交接见下方流程 |
| `references.md` | 只追加，不改已有条目，标注 `[执行者]` |
| `cross-refs.md` | 只追加，标注 `[执行者]` |
| `notation-registry.md` | 只追加，标注 `[执行者]` |
| `dependency-dag.md` | 核心依赖树由用户或指定 AI 维护；其他 AI 在更新日志区追加新发现的依赖（边列表格式），标注 `[执行者]` |
| `capability-assessment.md` | 只追加试跑结果，标注 `[执行者]` |
| `chapters/XX-*.md` | 只有 `status.md` 中标记为自己占用的才能写 |
| `methodology.md` | 只读，修改需用户在主终端讨论 |
| `outline.md` | 只读，修改需用户在主终端讨论 |

**关于共享文件在 status.md 中的表示**：`references.md`、`notation-registry.md` 等共享文件三家都可写入，在 status.md 中执行者列填 `共享`，不绑定某一家。

## 交叉验证机制

三家 AI 共存的最大优势是天然的交叉验证能力：

- **关键定理验证**：涉及核心结论的定理，由至少两家独立验证
- **事实核实**：历史事实、引用内容等，鼓励跨 AI 复核

### 章节审校流程

1. 原执行者完成初稿 → 更新 `status.md` 状态为 `初稿`
2. 任一其他 AI（或用户指派）认领审校 → 更新 `status.md`：状态改为 `审校中`，执行者改为 `原执行者 → 审校者`（如 `Claude → Gemini`）
3. 审校者读章节文件，审校意见写入 `chapters/XX-*.review.md`（新建文件，标注审校者名称如 `[Gemini]`）
4. 审校通过：审校者更新 `status.md` 状态为 `已完成`
5. 审校不通过：审校者更新 `status.md` 状态为 `修订中`，执行者改回原执行者

**审校对照清单**：`methodology.md` 第十一节（质量检查清单）

**与 methodology.md 阶段 4 的关系**：methodology.md 的"阶段 4：审校"是执行者自审清单（写完后自查）；此处的跨 AI 审校是自审**之后**的额外步骤。流程：原执行者自审 → 标 `初稿` → 另一家做交叉审校。

## 章节依赖与并行分配

### 低依赖章节（可以独立并行）
- 第 1-3 章（三段历史，彼此独立）
- 第 4-5 章 vs 第 9-10 章 vs 第 12-13 章（三大支柱各自的技术内容）

### 高依赖章节（需要前置章节完成）
- 第 16-19 章（交叉部分）依赖第二、三、四部分
- 第 8 章（一般理论）依赖第 4-7 章
- 第 20-22 章（应用）依赖第二部分

### 按支柱分配的参考方案（非强制）
```
支柱 A（发散级数）：第1章 → 第4-8章 → 第20章
支柱 B（哑运算）  ：第2章 → 第9-11章 → 第21章
支柱 C（拉马努金）：第3章 → 第12-15章 → 第22章
→ 汇合后：第16-19章（交叉部分），优先由已完成支柱的 AI 认领
```

具体哪家 AI 领哪条支柱，参考 `capability-assessment.md` 中的试跑结果和能力评估。未固定前任何一家可领任何章节。
